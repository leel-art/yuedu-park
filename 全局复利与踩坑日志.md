## 经验复利

> 记录开发中获得的有价值的经验、技巧、最佳实践。每一项都是复利，积累越多，开发越快。

### 通用经验

#### EXP-001 | 修改文件前先 Read
- **类别**：流程
- **内容**：在对任何文件做修改之前，必须先用 Read 工具完整读取文件内容，确认当前状态后再改。盲改极易引入错误。
- **复用场景**：每次代码修改

#### EXP-002 | 使用专用工具而非 Shell 命令操作文件
- **类别**：工具使用
- **内容**：读文件用 Read（非 `cat`/`Get-Content`），搜索用 Grep（非 `Select-String`），搜文件用 Glob（非 `Get-ChildItem -Recurse`）。专用工具更快且集成权限 UI。
- **复用场景**：日常开发

#### EXP-003 | 小改动用 Edit，大改动用 Write
- **类别**：工具使用
- **内容**：`Edit` 做精确替换（需先 Read），`Write` 做整文件重写。不要混用——用 `Write` 做小改动容易丢失其他内容。
- **复用场景**：每次文件编辑

#### EXP-004 | 破坏性操作前先确认
- **类别**：安全
- **内容**：涉及删除文件、重构架构、修改配置、force push 等操作，必须先向用户确认再执行。
- **复用场景**：Git 操作、文件删除、架构变更

#### EXP-005 | 一个 Commit 只做一件事
- **类别**：Git
- **内容**：不把多个不相关的改动混在一个 commit 里。回滚友好，review 清晰。
- **复用场景**：每次提交

### 小程序特定经验

#### EXP-MP-001 | setData 只传变更字段
- **类别**：性能
- **内容**：`this.setData({ key: newValue })` 而非 `this.setData(全量对象)`，避免不必要的渲染。
- **复用场景**：小程序页面/组件开发

#### EXP-MP-002 | data 中变量必须显式声明
- **类别**：规范
- **内容**：页面/组件的 `data` 对象中预先声明所有绑定变量，不要动态挂载。动态挂载的变量无法被 WXML 响应。
- **复用场景**：小程序页面/组件开发

## 回顾清单（Commit / 发布前自检）

每次提交或发布前，快速过一遍：

- [ ] 是否有 `console.log` 残留
- [ ] 是否有硬编码的测试数据或密钥
- [ ] 是否有未使用的变量 / 导入 / 死代码
- [ ] 资源路径是否正确（相对路径 / CDN）
- [ ] 错误处理是否完整（网络异常、空数据、边界条件）
- [ ] 基础库兼容性是否满足 ≥ 2.33.0
- [ ] 是否已在真机 / 模拟器上验证
